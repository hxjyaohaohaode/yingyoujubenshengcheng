# 极致优化白皮书

## 1. 本轮目标

本轮不是对外宣称“已达成军工级零故障”，而是先修复当前仓库中已经确认存在的确定性故障，建立后续大规模压测、容量规划、混沌工程与可重现构建的可信基线。

## 2. 已验证缺陷清单

### P0 已修复

1. **项目运行态字段读取错误**
   - 症状：`backend/api/ai.py`、`backend/api/export.py` 直接访问 `Project` 模型上不存在的 `genre/style/target_word_count/current_phase/core_truth` 字段。
   - 风险：导出、章节大纲生成、伏笔任务分发在运行时直接抛异常。
   - 处理：新增 `backend/services/project_runtime.py`，统一从 `Project + ProjectConfig` 组合出运行态视图。

2. **流水线状态 JSON 漂移**
   - 症状：`backend/core/pipeline/state_machine.py` 从数据库读取 `result_data/task_results/config` 时可能拿到字符串。
   - 风险：流水线执行器把字符串当 `dict/list` 使用，导致状态污染和后续步骤崩溃。
   - 处理：新增 JSON 容器强制归一化逻辑，异常载荷直接回退到安全默认值。

3. **任务进度仅存在进程内内存**
   - 症状：`backend/tasks/__init__.py` 使用 `TASK_PROGRESS` 字典保存进度。
   - 风险：多实例、进程重启、Worker 与 API 分离时任务状态丢失。
   - 处理：引入 Redis 持久化存储，保留内存兜底，默认 TTL 1 小时。

4. **WebSocket 事件协议不一致**
   - 症状：后端发送 `agent_update`，前端只监听 `agent_status`；前端还错误要求 `task_progress` 必须带 `agent_name`。
   - 风险：Agent 状态面板不刷新，任务进度消息被静默丢弃。
   - 处理：后端统一发送 `agent_status`，前端同时兼容新旧事件名，并允许无 `agent_name` 的任务进度消息。

5. **后端 Dockerfile 沿用过期目录结构**
   - 症状：镜像构建复制不存在的 `tasks.py`、`agents/`。
   - 风险：容器构建失败或运行时缺核心模块。
   - 处理：改为复制真实目录 `core/`、`migrations/`、`templates/`，删除过期路径。

6. **仓库混入本地产物与构建垃圾**
   - 症状：仓库内存在 `dist`、`__pycache__`、`script_engine.db`、`tsbuildinfo` 等本地生成文件。
   - 风险：环境漂移、仓库膨胀、提交污染、构建上下文冗余。
   - 处理：新增 `.gitignore` 与 `.dockerignore`，并清除现有本地产物。

7. **缺少持续集成门禁**
   - 症状：修改后没有统一的自动化门禁验证。
   - 风险：局部修复回归难以及时发现。
   - 处理：新增 GitHub Actions CI，自动执行后端测试、后端编译与前端构建。

8. **任务进度缺少 Agent 元数据**
   - 症状：任务链路虽然能推送进度，但很多事件不带 `agent_name/task_name`。
   - 风险：前端难以精确展示是谁在执行什么。
   - 处理：为核心 Celery 任务统一补充 Agent 与任务名称元数据。

## 4. 验证结果

### 已通过

- `python -m unittest backend.tests.test_project_runtime backend.tests.test_pipeline_state_machine backend.tests.test_task_progress -v`
- `python -m unittest discover -s backend/tests -p "test_*.py" -v`
  - 结果：12 个回归测试全部通过。

- `npm run build`
  - 结果：前端生产构建通过。

- `python -m compileall backend`
  - 结果：后端源码编译通过。

### 已发现但未关闭

- 前端产物仍存在大体积 chunk，`vendor-antd` 仍超过 1 MB，需要进一步按组件或按路由拆包。
- 仓库尚未建立完整 contract test、CI、混沌工程、容量压测、第三方审计流程。
- 尚未形成 150 万字剧本的真实性能基线，因此不能声称达成 5 秒加载、P99 200ms、CPU < 60% 等指标。

## 5. 可量化收益

### 本轮已实证

- 消除了 3 条明确的运行时属性错误路径。
- 将任务进度从“单进程内存态”升级为“Redis 持久化优先 + 内存兜底”。
- 将实时协议从“前后端不兼容”修复为“双向兼容”。
- 使后端镜像构建脚本重新对齐当前代码结构。
- 建立了可执行的 CI 门禁骨架。
- 清除了当前仓库中的本地构建产物和缓存目录。
- 将 `ScriptViz` 相关前端 chunk 从原先约 164 kB 压缩到约 34 kB 级别，但 `antd` 主包仍偏大。

### 本轮未实证，不得宣称

- 可用性达到 99.99%
- 150 万字全链路耗时降低到基线 20%
- 仓库体积缩减 30%
- 构建速度提升 50%
- 所有用户可见功能 contract coverage 95%
- 零高危、零后门第三方审计结论

## 6. 回滚方案

### 单文件级回滚

- 若运行态配置适配异常，回滚：
  - `backend/services/project_runtime.py`
  - `backend/api/ai.py`
  - `backend/api/export.py`

- 若任务状态读取异常，回滚：
  - `backend/tasks/__init__.py`

- 若实时面板出现兼容问题，回滚：
  - `backend/websocket/manager.py`
  - `frontend/src/hooks/useWebSocket.ts`

- 若镜像构建流程异常，回滚：
  - `backend/Dockerfile`

### 数据安全说明

- 本轮未执行破坏性迁移。
- 新增 Redis 任务进度键为短期缓存，不影响主业务表数据。
